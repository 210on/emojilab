# デザインスコアの仕組み: UI実例から分解する

このページは、EmojiLab. の `デザインスコア` が UI 上でどのように表示され、内部でどのように計算されているかを、実例から一段ずつ分解して説明するための資料である。

対象読者は、研究発表・論文執筆時に「このスコアは何を測っているのか」「どこまで客観的に言えるのか」「どこからがヒューリスティックなのか」を説明する必要がある人を想定する。

## 0. 前提

デザインスコアは、カスタム絵文字の「良し悪し」を完全に決める絶対評価ではない。目的は、制作中のユーザーに対して、見やすさを損ないやすい条件を即時に返すことである。

したがって、論文では次のように扱う。

```text
EmojiLab. のデザインスコアは、APCA コントラスト、縮小耐性、構成安定性を統合したルールベースの設計支援指標である。
これは人間の視認性を直接測定する標準尺度ではなく、制作支援のための操作的指標として設計し、人間評価との対応を検証する対象である。
```

## 1. UI上で見えているもの

デザインスコアカードには、主に次の要素がある。

```text
デザインスコア

[円形リング] 95

APCAコントラスト   Lc 80
[バー]

縮小耐性            94%
[バー]

黄系の塗りと線の分離が良く、小サイズでも視認性を保ちやすい構成です。
```

この UI は、内部的には次の対応になっている。

| UI要素 | 内部値 | 役割 |
|---|---:|---|
| 円形リングの数値 | `designScore.total` | 総合支援スコア |
| APCAコントラスト | `displayedContrastLc` | 最も弱い代表コントラスト |
| 縮小耐性 | `scalabilityScore` | 小サイズ表示で潰れにくいか |
| 改善コメント | `designFeedbackService` | 最も優先度の高い改善観点 |

### UI上の色変化

UIでは、数値だけでなく色でも状態を示す。

| UI要素 | 緑 | 黄 | 赤 |
|---|---|---|---|
| 円形リング | `total >= 80` | `70 <= total < 80` | `total < 70` |
| APCAコントラストバー | `Lc >= 75` | `60 <= Lc < 75` | `Lc < 60` |
| 縮小耐性バー | `scalabilityScore >= 82` | `72 <= scalabilityScore < 82` | `scalabilityScore < 72` |

このため、円形リングは総合点の直感的な状態表示として読みやすい。一方で、APCAまたは縮小耐性が赤域にある場合は `conditionCap` により総合点自体が80未満へ制限されるため、赤い個別指標があるのにリングだけ緑になる矛盾を避ける。

### UI表示の実例

| 例 | 総合リング | APCAバー | 縮小耐性バー | UI上の読み方 |
|---|---|---|---|---|
| `あり / がと` 黄色 + 黒内側 + 白外側 | 緑 `95` | 緑 `Lc 80` | 緑 `94%` | 総合・コントラスト・縮小耐性がすべて良好 |
| 黄色、内側線なし | 赤 `63` | 赤 `Lc 16` | 緑 `91%` | 文字構成は保てるが、背景/局所コントラストが致命的 |
| `確認 / 鬱憤` 太字 + 内側線 | 黄 `78` | 緑 `Lc 80` | 赤 `46%` | コントラストは十分だが、高画数文字で縮小耐性が低い |

## 2. 実例: 初期状態の「ありがと」

現行の初期設定に近い例を使う。

```text
上: あり
下: がと
塗り: #FFCC00
内側線: #000000 / 4
外側線: #FFFFFF / 10
フォント: LINE Seed JP
ウェイト: 700 (Bold)
横幅: 100
行間: -50
背景: Light #FFFFFF / Dark #1D1C1D / Custom #EEF3FA / #2B2D31
```

この場合、現行実装では代表値は次のようになる。

| 指標 | 値 | 読み方 |
|---|---:|---|
| 総合スコア | 95 | 良好 |
| コントラスト適合 | 95 | 塗り・線・背景の分離が強い |
| 表示 APCA | Lc 80 | 最も弱い代表ペアでも十分高い |
| 縮小耐性 | 94 | 文字数・線幅・ウェイトの条件が安定 |
| 構成安定性 | 93 | 2行の塊が正方形領域に近い |

ここで重要なのは、黄色そのものは白背景に弱いという点である。実装上、`#FFCC00` と白背景の直接コントラストは弱い。しかし、黒い内側線が塗り色との局所コントラストを作り、白い外側線がダーク背景上で輪郭を作るため、総合的には高評価になる。

## 3. ステップ1: まず色と線を分けて見る

文字絵文字は、単純な「文字色 vs 背景色」では評価しにくい。

EmojiLab. では、見た目の層を次のように扱う。

```text
塗り
内側線
外側線
背景
```

この層ごとに、次のような関係を評価する。

```text
塗り vs 内側線
塗り vs 外側線
塗り vs 背景
内側線 vs 背景
外側線 vs 背景
```

ただし、線が細すぎる場合は、APCA 値が高くても視覚的な境界として機能しない可能性がある。そのため、現行実装では次の有効判定を持つ。

```text
内側線が有効:
  stroke1Enabled && stroke1Width >= 2

外側線が有効:
  stroke2Enabled && stroke2Width >= 4

外側線が安定:
  stroke2Enabled && 8 <= stroke2Width <= 14
```

この処理により、たとえば `外側線 1px` のような設定が、数値上の色差だけで過大評価されることを避ける。

## 4. ステップ2: APCAコントラストを代表値にする

EmojiLab. では APCA を使って、前景と背景の明暗差を評価する。

APCA は、色差を単純な比率ではなく知覚コントラスト寄りに扱う指標であり、文字サイズやウェイトなどの空間特性が知覚に影響するという前提に立つ。APCA の説明では、線幅・文字ウェイト・サイズといった spatial characteristics が知覚コントラストに関係することが示されている。

参考: [APCA Easy Intro](https://git.apcacontrast.com/documentation/APCAeasyIntro)

一方、WCAG 2.2 の Contrast Minimum は、通常テキストで `4.5:1`、大きなテキストで `3:1` を要求する。これは Web テキストのアクセシビリティ基準として有用だが、カスタム絵文字のような小さな画像文字では、色の印象を保つことや縁取りによる局所コントラスト補助も重要になる。

参考: [WCAG 2.2 Success Criterion 1.4.3 Contrast Minimum](https://www.w3.org/TR/WCAG22/#contrast-minimum)

EmojiLab. では、UI に表示する APCA は次のように決める。

```text
localTextLc =
  max(
    APCA(fill, innerStroke),
    APCA(fill, outerStroke),
    APCA(fill, background) if no effective stroke
  )

backgroundSeparationLc =
  min over backgrounds:
    max(
      APCA(fill, background),
      APCA(innerStroke, background) if inner effective,
      APCA(outerStroke, background) if outer effective
    )

displayedContrastLc =
  min(localTextLc, backgroundSeparationLc)
```

つまり、UI の `Lc 80` は、単純な `塗り色 vs 背景色` ではない。塗り、内側線、外側線、複数背景を考慮したうえで、最も弱い代表箇所を表示している。

## 5. ステップ3: 総合点用の内部スコアに変換する

APCA の参照値は `displayedContrastLc` として UI に残す。一方、総合スコアでは縮小耐性や構成安定性と合成する必要があるため、`localTextLc` と `backgroundSeparationLc` を 0-100 の内部点 `contrastFitScore` に変換する。

現行実装では次のように計算する。

```text
contrastFitScore =
  normalizeLc(localTextLc) * 0.55
  + normalizeLc(backgroundSeparationLc) * 0.45
```

ここで重要なのは、`contrastFitScore` は APCA そのものではないという点である。論文やUIで APCA として参照する値は `displayedContrastLc` / `Lc` であり、`contrastFitScore` は総合支援スコアに合成するための派生指標である。

UIの `Lc` とコメント分岐では、概念的に次の帯域で扱う。

| Lc帯域 | 扱い |
|---:|---|
| `Lc < 45` | 改善優先 |
| `45 <= Lc < 60` | 注意 |
| `60 <= Lc < 75` | 許容 |
| `Lc >= 75` | 良好 |

ただし、コントラストが高ければ常に良いわけではない。たとえば黒い内側線が不要な色に太く入っている場合、文字内部を複雑にし、小サイズで潰れやすくなる。その場合、コントラスト側では読めても、縮小耐性側で減点される。

## 6. ステップ4: 縮小耐性を見る

縮小耐性は、Slack や Discord のリアクション、本文内絵文字、モバイル表示など、小さい表示で文字形が保たれるかを推定する。

評価対象は次のとおり。

| 要素 | 影響 |
|---|---|
| 文字数 | 多いほど潰れやすい |
| 漢字画数 | 高画数ほど内部空間が詰まりやすい |
| フォントウェイト | 太すぎると高画数文字で潰れやすい |
| 内側線 | 色補助になるが、太いと字面を侵食する |
| 外側線 | 輪郭を作るが、太すぎると面積を圧迫する |
| 横幅補正 | 強い圧縮・伸張は字形判別性を落とす |
| 行間・字間 | 詰まりや空きすぎを作る |

画数は Unicode Unihan の `kTotalStrokes` に基づく辞書を用いる。現行辞書は `kJoyoKanji`、`kJinmeiyoKanji`、`kJis0`、`kJis1`、`kJIS0213` を含み、13,110字を収録している。

現行実装の画数ペナルティは、概念的には次のように働く。

| 条件 | 評価 |
|---|---|
| 最大画数 14以上 | 注意 |
| 最大画数 16以上 | 減点 |
| 最大画数 20以上 | 強い減点 |
| 高画数漢字が複数 | 追加減点 |
| 高画数漢字 + 太ウェイト | 追加減点 |
| 高画数漢字 + 太い内側線 | 追加減点 |

例として、`確認 / 鬱憤` のように高画数漢字を含めると、コントラストが良くても縮小耐性が下がる。

| 例 | 総合 | Lc | 縮小耐性 | 主な理由 |
|---|---:|---:|---:|---|
| `あり / がと` 黄色 + 黒内側 + 白外側 | 95 | 80 | 94 | 色と線の分離が安定 |
| `確認 / 鬱憤` 同設定 | 78 | 80 | 46 | 高画数漢字 + 太ウェイト + 線で詰まりやすい |

## 7. ステップ5: 構成安定性を見る

構成安定性は、絵文字全体が正方形領域を効率よく使えているかを弱く評価する。

Slack はカスタム絵文字として正方形画像を推奨しており、透明背景の正方形画像が適していると案内している。

参考: [Slack: Add custom emoji and aliases](https://slack.com/help/articles/206870177-Add-custom-emoji-and-aliases-to-your-workspace)

Discord もカスタム絵文字にファイルサイズや名称制約を設けている。

参考: [Discord: How to Add Custom Emojis](https://support.discord.com/hc/en-us/articles/360036479811-How-to-Add-Custom-Emojis-on-Discord)

EmojiLab. では、正方形性そのものを強い正解にはしない。文字の意味や表現意図を壊してまで正方形にするべきではないためである。

ただし、極端に縦長・横長になると、チャット上の限られた表示領域を使い切れない。そのため、構成安定性では次のようなリスクを見る。

```text
横幅補正リスク
行間リスク
上下行の幅バランスリスク
縦横比リスク
```

例として、`THANK / YOU!!` のように文字数が多く、横幅を 82% まで圧縮した場合、コントラストは高くても構成安定性と縮小耐性が下がる。

| 例 | 総合 | コントラスト | 縮小耐性 | 構成 |
|---|---:|---:|---:|---:|
| `あり / がと` | 95 | 95 | 94 | 93 |
| `THANK / YOU!!` 横幅82 | 89 | 95 | 78 | 85 |

## 8. ステップ6: 総合スコアに統合する

最終的なデザインスコアは、加点ではなく `100点から視認性リスクを減点する` 方式で計算する。

この設計にした理由は、カスタム絵文字では「良い要素の合計」よりも、「読めなくなる原因が残っていないか」が重要だからである。コントラストまたは縮小耐性のどちらかが弱い場合、他の指標が高くても見づらさが残るため、致命的リスクとして追加減点する。

```text
contrastRisk =
  (100 - contrastFitScore) * 0.35

scalabilityRisk =
  (100 - scalabilityScore) * 0.30

compositionRisk =
  (100 - compositionScore) * 0.15

criticalRisk =
  displayedContrastLc < 45 なら 15
  45 <= displayedContrastLc < 60 なら 8
  scalabilityScore < 60 なら 15
  60 <= scalabilityScore < 72 なら 8

designScore =
  min(rawScore, conditionCap)

rawScore =
  100 - contrastRisk - scalabilityRisk - compositionRisk - criticalRisk

conditionCap =
  displayedContrastLc < 45 または scalabilityScore < 60 なら 69
  displayedContrastLc < 60 または scalabilityScore < 72 なら 79
  それ以外は 100
```

減点幅の意図は次のとおり。

| 減点群 | 最大寄与 | 理由 |
|---|---:|---|
| コントラストリスク | 35% | 文字と背景・線の分離は視認性の基礎 |
| 縮小耐性リスク | 30% | カスタム絵文字は小サイズ表示が多い |
| 構成リスク | 15% | 正方形領域の利用効率と過度な変形を補助的に見る |
| 致命的リスク | 最大30点 | APCAまたは縮小耐性が閾値未満の場合に相殺を防ぐ |
| 条件上限 | 69 / 79 / 100 | 赤表示の公開指標がある場合に、総合点が80点台へ入ることを防ぐ |

この値は、研究開始時点の初期ヒューリスティックである。実験では、人間評価との対応を見て、減点幅・閾値・コメント分岐の妥当性を検証する。

例えば `馬鹿 / 鬱病` のように4文字かつ高画数漢字を含む入力では、APCA が高くても縮小耐性が赤域に入ることがある。この場合、総合点が80点以上に見えると「ほぼ合格」と解釈されやすいため、総合点は最大79に制限する。

## 9. ステップ7: ステータス表示に変換する

UI 上のラベルと円形メーター色は、最終的な総合点の帯域で決める。

```text
80以上: 良好 / 緑
70-79: 調整推奨 / 黄
69以下: 要改善 / 赤
```

この方が、ユーザーは `80点以上はほぼ合格`、`70点台は直す余地あり`、`69点以下は改善が必要` と直感的に読める。

ただし、APCA または縮小耐性が赤域にある場合は `conditionCap` により総合点を80未満へ抑える。これにより、赤い個別指標があるのに総合点だけ緑になる矛盾を避ける。

## 10. ステップ8: 改善コメントを作る

改善コメントは、すべての指標を説明するものではない。ユーザーが次に直すべき一点を返す。

現行実装では、固定順序だけでコメントを決めない。まず改善候補を列挙し、それぞれに推定インパクトを与える。そのうえで、最も総合スコアの上昇幅が大きいと見込まれる候補を選ぶ。

```text
候補:
  コントラスト不足
  背景分離不足
  高画数漢字 + 太ウェイト
  内側線過多
  外側線不足
  縦横比の偏り

選択:
  推定改善インパクトが最大の候補
```

つまり、コメントは一般的な総評ではなく、「ここを直すと一番効く可能性が高い」という設計誘導である。

### 現行の改善コメント候補

現行実装では、主に次の候補文から1つを表示する。

推定インパクトは、実際に修正後のデザインを生成して再採点するものではない。現在値から「不足量」「直接減点されている量」「一手で直せる可能性」を近似している。

| 候補ID | 主な発火条件 | 推定インパクトの計算 | 表示文言 | 促す操作 |
|---|---|---|---|---|
| `missing-inner-contrast-support` | 明るい高彩度色が白背景で埋もれ、内側線が十分に効いていない | `(75 - min(displayedContrastLc, 75)) × 0.45 + 10` | 明るい塗り色が白背景で埋もれやすいので、黒など十分に差のある内側線を少し足すと色味を保ったまま読みやすくなります。 | 内側線を有効化し、黒または十分に差のある色を細く足す |
| `weak-local-contrast` | 塗りと線の局所APCAが45未満 | `(45 - localTextLc) × 0.6 + 16` | 塗りと線の差が弱いので、塗り色か内側線のどちらかを十分に離すと文字形が読み取りやすくなります。 | 塗り色または内側線色を離す |
| `weak-background-separation` | 背景分離APCAが45未満 | `(45 - backgroundSeparationLc) × 0.55 + 14` | 背景との分離が弱いので、外側線を少し強めるとチャット背景上で輪郭が保ちやすくなります。 | 外側線を有効化または太くする |
| `dense-kanji-heavy-weight` | 最大画数16以上かつウェイト800以上 | `(100 - scalabilityScore) × 0.35 + 10` | 画数の多い漢字に太いウェイトが重なり、小サイズで詰まりやすいです。太さか内側線を少し弱めると安定します。 | フォントウェイトを下げる、または内側線を弱める |
| `dense-kanji-complexity` | 最大画数16以上かつ縮小耐性が72未満 | `(72 - scalabilityScore) × 0.5 + denseKanjiCount × 4 + 8` | 画数の多い漢字が小サイズで詰まりやすいです。画数の少ないシンプルな表現にするか、太さ・内側線・字間を少し軽くすると安定します。 | 表現の簡略化、フォントウェイト、内側線、字間を見直す |
| `inner-stroke-too-heavy` | 内側線が太い、または不要な内側線リスクがある | `(stroke1Width - 8) × 2 + unnecessaryInnerStrokeRisk ? 10 : 0` | 内側線が強く、塗りの面積を圧迫しています。内側線を細くすると小サイズでの抜けが良くなります。 | 内側線幅を下げる |
| `dense-kanji-inner-stroke` | 高画数漢字があり、内側線が8以上 | `denseKanjiCount × 4 + stroke1Width` | 画数の多い漢字に内側線が重く、小サイズで詰まりやすいです。内側線を少し細くすると安定します。 | 内側線幅を下げる |
| `outer-stroke-too-thin` | 外側線が無効、または細すぎる | `outer invalid ? 12 : 8 + max(0, 60 - displayedContrastLc) × 0.25` | 外側線が輪郭として効きにくいので、少し太くするとチャット背景上で形を保ちやすくなります。 | 外側線を太くする |
| `composition-aspect` | 全体の縦横比が大きく偏る | `(100 - compositionScore) × 0.2 + 8` | 全体の縦横比が偏っているので、横幅や幅揃えを調整すると表示領域をより効率よく使えます。1段だけの場合は、下段にも分ける提案を優先する。 | 横幅、幅揃え、上下サイズ比、段分けを調整する |
| `general-scalability` | 縮小耐性が72未満で、単一原因に分解しきれない | `(72 - scalabilityScore) × 0.35 + 8` | 縮小耐性が不足しています。内側線を細くし、字間を少し広げるか、横幅を100に近づけると小サイズで安定しやすくなります。 | 内側線、字間、横幅を見直す |

最終的には、`impact` が最も大きい候補を選ぶ。ただし、危険域でも非許容域でもなく、最大インパクトが小さい場合は、強い改善指示を出さず、許容域・良好域のコメントに切り替える。

許容域に入っている場合は、強い改善コメントではなく、次のような軽い調整または肯定コメントに切り替える。

| 状態 | 表示文言 |
|---|---|
| 内側線を弱める余地がある | 内側線がやや強く、色面を圧迫しています。内側線を少し弱めると、色味を保ちながらよりすっきり見せられます。 |
| 高画数漢字だが許容域 | このままでも読めますが、高画数の漢字は太さを少し軽くすると小サイズでさらに安定します。 |
| 調整推奨域 | 最も弱い指標に応じて、外側線、内側線・字間、横幅・幅揃えのいずれかを具体的に提示する。 |
| 良好 | 黄系の塗りと線の分離が良く、小サイズでも視認性を保ちやすい構成です。 |
| 良好、一般 | コントラスト、縮小耐性、構成のバランスが良く、色味を保ったまま十分見やすい仕上がりです。 |

例:

| 状態 | コメントの方向 |
|---|---|
| 明るい黄色が白背景で埋もれる | 黒い内側線を少し足す |
| 青系で黒内側線が太い | 内側線を弱めて複雑さを減らす |
| 高画数漢字 + 太ウェイト | ウェイトまたは内側線を弱める |
| 外側線が細い | 背景から輪郭を分離するため外側線を太くする |
| 縦横比が偏る | 横幅・幅揃えを調整する |

## 11. 逆転を避けるための設計

避けたい問題は、次のような評価の逆転である。

```text
高スコアなのに強い改善コメントが出る
低スコアなのに肯定コメントが出る
コントラストが十分なのに不要な黒内側線を推奨する
縮小耐性が低いのに「見やすい」と言う
```

現行設計では、コメント生成前に次の帯域を判定する。

```text
danger:
  displayedContrastLc < 45
  or scalabilityScore < 60
  or total < 60

acceptable:
  displayedContrastLc >= 60
  and scalabilityScore >= 72
  and total >= 70

good:
  total >= 80
```

この帯域判定により、スコアとコメントのトーンを揃える。

## 12. 研究上の限界

このスコアは、以下をまだ直接測っていない。

- 実際のチャット画面上での視線移動
- 個人差のある色覚・視力・表示環境
- 文字の意味理解や文脈適合
- 使用者がその絵文字を「使いたい」と感じるか
- すべてのフォントでの字形差

したがって、研究では次のように述べるのが妥当である。

```text
本システムは、カスタム絵文字制作時に発生しやすい視認性上のリスクを、コントラスト・縮小耐性・構成安定性の観点から即時に提示する。
スコアは制作支援のための操作的指標であり、その妥当性は第三者評価・文字転記課題・制作支援比較実験によって検証する。
```
