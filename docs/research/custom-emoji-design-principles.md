# カスタム絵文字のデザイン要素と設計指針

このページは、EmojiLab. で扱うカスタム絵文字のデザイン要素を、制作手順に沿って一段ずつ整理したものである。

目的は、研究・発表・論文で次の問いに答えられるようにすることである。

- カスタム絵文字では、どのデザイン要素が視認性に影響するのか
- 色、枠線、フォント、ウェイト、文字量、縦横比をどう扱うべきか
- なぜ黒白最大コントラストだけを正解にしないのか
- EmojiLab. のデザインスコアと改善コメントは、どの指針に基づくのか

## 0. カスタム絵文字の特殊性

カスタム絵文字は、通常の本文テキストと違い、次の特徴を持つ。

| 特徴 | 意味 |
|---|---|
| 画像として投稿される | 書体・色・枠線が固定された小さな画像になる |
| 表示サイズが小さい | リアクションや本文中では 16px〜32px 程度で見られる |
| 背景が変わる | ライトテーマ、ダークテーマ、Slack風、Discord風、カスタム背景で見え方が変わる |
| 表現意図が強い | 色や形が、意味・感情・コミュニティ文化を担う |
| 正方形領域で扱われる | 極端な縦長・横長は表示面積を無駄にする |

そのため、カスタム絵文字では「読めること」と「色や雰囲気が残ること」を同時に扱う必要がある。

## 1. 最初に決めること: 何を伝える絵文字か

最初に決めるのは、見た目ではなく用途である。

```text
例:
ありがとう
確認
了解
天才
つらい
最高
修正お願いします
```

用途によって、許容できる文字量と色の方向が変わる。

| 用途 | 推奨 |
|---|---|
| 反応・リアクション | 1〜4文字程度が望ましい |
| 依頼・状態表示 | 2行構成を使う |
| 感情表現 | 色の印象を優先してよい |
| 業務連絡 | 可読性を優先する |

設計上は、最初に次の問いを置く。

```text
この絵文字は、誰が、どのチャット上で、どのサイズで、何秒以内に読めればよいか。
```

## 2. 文字量: 少ないほど強い

カスタム絵文字は表示サイズが小さいため、文字数が増えるほど視認性は落ちる。

目安:

| 文字量 | 扱い |
|---|---|
| 1〜2文字 | 最も安定 |
| 3〜4文字 | 推奨範囲 |
| 5〜6文字 | 2行構成・幅揃えが必要 |
| 7文字以上 | 読ませるより雰囲気表現に近くなる |

例:

```text
良い:
上: 確認
下: よろしく

弱い:
上: 確認お願いします
下: よろしくお願いします
```

EmojiLab. では、文字数が増えると `縮小耐性` が下がる。これは、小サイズで一文字あたりに使える面積が減るためである。

## 3. 漢字と画数: 意味は強いが潰れやすい

漢字は短い文字数で意味を伝えられる。一方で、画数が多い漢字は小サイズで内部空間が潰れやすい。

例:

```text
低リスク:
大
小
中
了
天

注意:
確
認
優
議

高リスク:
鬱
鑑
驚
魔
```

EmojiLab. では、Unicode Unihan の画数データを使い、最大画数・合計画数・高画数漢字数を縮小耐性に反映している。

目安:

| 最大画数 | 扱い |
|---:|---|
| 13画以下 | 通常 |
| 14〜15画 | 注意 |
| 16〜19画 | 小サイズで詰まりやすい |
| 20画以上 | 強い注意 |

高画数漢字を使う場合の指針:

- フォントウェイトを少し下げる
- 内側線を太くしすぎない
- 文字数を減らす
- 行間を詰めすぎない
- 横幅圧縮を強くしすぎない

## 4. 色: 黒白最大コントラストだけを正解にしない

黒と白は最も分離しやすい。しかし、すべてのカスタム絵文字を黒白だけで作ると、意味・感情・用途に応じた表現が弱くなる。

EmojiLab. では、色を次の2つの役割で扱う。

| 役割 | 内容 |
|---|---|
| 視認性 | 背景や線と十分に分離しているか |
| 表現意図 | 黄色、赤、青、緑などの印象が残っているか |

たとえば黄色は、白背景では弱い。しかし、黄色を暗くして WCAG AA に近づけると、黄色らしい明るさが失われやすい。

この場合、EmojiLab. では次の考え方を取る。

```text
塗り色そのものを濁らせるのではなく、必要な場合だけ内側線で局所コントラストを補う。
```

これは「色の印象」と「読める輪郭」を分離して扱う方針である。

## 5. PCCS bright tone を使う理由

現行パレットは PCCS bright tone を基準にしている。

理由は次のとおり。

| 理由 | 説明 |
|---|---|
| 色相を体系的に提示できる | Hue-Tone Systemの同一トーンカテゴリから12色相を採るため、刺激条件を再生成できる |
| 恣意的な選色を減らせる | 任意の流行色や制作者の好みではなく、PCCS記号で初期条件を説明できる |
| 色相と背景適合を分けて扱える | 色相探索はPCCS、実際の背景コントラストはAPCAという役割分担が明確になる |
| 人間評価で検証できる | 同じ12色を用いて色相、背景、線の効果を反復可能な条件として比較できる |

ただし、PCCS bright tone は、すべての色が白背景・黒背景に対して同じコントラストになるようには設計されていない。現行12色の実測範囲は、白背景で APCA `Lc 8-76`、黒背景で `Lc 25-97` である。そのため、色相ごとの背景適合差は残る。

「bright toneは絵文字に最適」「vivid toneより眩しくない」とは、現段階では主張しない。bright toneの採用は色相表現を保った初期探索条件であり、最適性は認識成績・反応時間・主観評価によって検証する必要がある。

論文では次のように記述するのが妥当である。

```text
本研究では、色相間の比較可能性を確保するため、PCCS bright tone を初期パレットとして採用した。
ただし、同一トーンであっても背景との明度差は色相によって異なるため、コントラスト評価と縁取り提案を併用する。
```

実測値、主張可能範囲、検証仮説は [デフォルトカラーセットの選定根拠](./default-color-palette-rationale.md) にまとめる。

## 6. 内側線: 色を残すための補助

内側線は、塗り色と文字の輪郭を分離するために使う。

特に有効なのは、次の条件である。

```text
明るい塗り色
高彩度
白背景で埋もれやすい
文字形の内部を少し補助すれば読める
```

例:

| 塗り色 | 内側線の扱い |
|---|---|
| 黄・黄緑・橙 | 黒内側線が有効になりやすい |
| 青・紫 | 黒内側線が不要または過剰になりやすい |
| 赤・桃 | 背景や明度によって判断 |
| 白系 | 黒内側線がほぼ必要 |
| 黒系 | 白内側線または外側線を検討 |

重要なのは、色名で固定しないことである。

```text
「b6, b8, b10 だから黒内側線」ではなく、
「塗り色が対象背景で弱く、内側線が十分な局所コントラストを作れるから黒内側線」
と説明する。
```

現行パレットでは、そのルールの結果として黄・黄緑・橙が該当しやすい。

## 7. 内側線を付けない方がよい場合

内側線は常に良いわけではない。

付けない、または弱めた方がよい条件:

- 塗り色が白背景で十分に読める
- 青・紫・濃色など、塗り色自体の明度差が大きい
- 画数の多い漢字を使っている
- フォントウェイトが重い
- 内側線が塗り面積を圧迫している

例:

```text
青い塗り + 黒内側線 8
```

この場合、コントラストは十分でも、文字内部が黒く詰まり、小サイズで読みづらくなる可能性がある。EmojiLab. では、このような状態を `縮小耐性` や `改善コメント` で扱う。

## 8. 外側線: 背景から切り離すための輪郭

外側線は、絵文字全体を背景から分離するために使う。

特に重要なのは、ライトテーマとダークテーマの両方で使われる点である。

```text
ライト背景:
  黒・濃色・彩度の高い塗りは読みやすいが、白外側線は見えにくい

ダーク背景:
  黄色・白・明色は読みやすいが、黒内側線だけでは輪郭が沈む場合がある
```

そのため、EmojiLab. の初期設定では、外側線に白を使う。これは、ダーク背景上で絵文字の外周を切り離すためである。

外側線の目安:

| 幅 | 扱い |
|---:|---|
| 0〜1 | ほぼ境界として扱わない |
| 2〜3 | 効く場合もあるが不安定 |
| 4〜7 | 有効だが細め |
| 8〜14 | 安定範囲 |
| 15〜20 | 強い表現 |
| 21以上 | 面積圧迫・複雑化に注意 |

現行の推奨は、まず `10` 前後から始めることである。

## 9. フォント: 太ければよいわけではない

カスタム絵文字では太いフォントが有利になりやすい。小サイズでも線が残るためである。

しかし、太すぎるフォントは高画数漢字では逆効果になる。

| 文字条件 | 推奨 |
|---|---|
| ひらがな・カタカナ中心 | 700〜900 が使いやすい |
| 漢字少なめ | 700〜900 |
| 高画数漢字あり | 600〜800 程度も検討 |
| 明朝系 | 太さより字形の抜けを確認 |
| 丸ゴシック | 柔らかいが内部空間が詰まる場合がある |

現行 UI では、フォントが対応しているウェイトだけを選べるようにしている。対応していないウェイトを無理に指定すると、ブラウザが別フォントや合成ウェイトにフォールバックし、見た目が変わるためである。

## 10. 横幅補正: 正方形に近づけるが、潰しすぎない

カスタム絵文字は正方形領域で表示されるため、2行の文字幅が大きく違う場合は、横幅補正や幅揃えが有効である。

ただし、横幅補正は字形を変形させる。強く圧縮・伸張しすぎると、文字の骨格が崩れ、読みづらくなる。

目安:

| 横幅 | 扱い |
|---:|---|
| 90〜110 | 自然 |
| 85〜115 | 許容 |
| 80〜120 | 強いが使用可能 |
| 80未満 / 120超 | 注意 |

幅揃えの目的は、両行を完全に同じ幅にすることではない。可読性を壊さない範囲で、絵文字全体の塊を安定させることである。

## 11. 行間・字間: 詰めればよいわけではない

2行構成では、行間を詰めると大きく見える。しかし、詰めすぎると上下の線や塗りが干渉する。

目安:

| 設定 | 影響 |
|---|---|
| 行間を詰める | 全体が大きく見えるが、上下干渉リスク |
| 行間を広げる | 読みやすいが、絵文字全体が小さくなる |
| 字間を詰める | まとまりやすいが、判別しにくい |
| 字間を広げる | 判別しやすいが、横長になりやすい |

EmojiLab. では、塗り・内側線・外側線を含む外接矩形が重ならないように描画している。これは、行間を詰めても下段の線が上段の塗りを覆いにくくするためである。

## 12. 上下サイズ比: 2行の意味を整理する

上下サイズ比は、2行のどちらを主役にするかを決める。

例:

```text
上: 確認
下: よろしく
```

この場合、`確認` を大きくすると、意味の中心が明確になる。一方、`よろしく` を大きくすると、依頼・挨拶のニュアンスが強くなる。

指針:

- 意味の核を大きくする
- 長い行を無理に大きくしない
- 2字 + 5字のような構成では、短い行を大きくして視覚的な幅を近づける
- サイズ比を変えた後に、幅揃えと横幅を再調整する

## 13. 背景プレビュー: ライトとダークを両方見る

カスタム絵文字は、作った画面と使われる画面が一致しない。

そのため、最低限次の背景で確認する。

| 背景 | 見ること |
|---|---|
| 白・ライトグレー | 明るい塗りが埋もれないか |
| 暖色ライト | 白外側線が弱すぎないか |
| 寒色ライト | 青系・水色系が沈まないか |
| チャコール・ダーク | 黒内側線だけで沈まないか |
| Discord/Slack風 | 実利用に近いなじみ方 |

ライトだけで良い絵文字は、ダークで輪郭が消えることがある。ダークだけで良い絵文字は、ライトで塗りが埋もれることがある。

## 14. チャットプレビュー: 単体表示と文中表示を見る

カスタム絵文字は、リアクションだけでなく、単独投稿や本文中にも使われる。

確認すべき状態:

- リアクションとして小さく表示した状態
- 本文中に混ざった状態
- 単独で投稿された状態
- Slack風の背景
- Discord風の背景

この確認により、制作画面では見えていた線や細部が、実利用サイズで消える問題を見つけやすくなる。

## 15. PNG書き出し: 透明余白を避ける

カスタム絵文字の出力では、文字と枠線の周囲に不要な透明余白があると、実際の表示サイズが小さくなる。

EmojiLab. では、透明ピクセルをトリミングし、絵文字の不透明領域に合わせて書き出す。

指針:

- 外側線が欠けない
- 透明余白を残しすぎない
- 正方形領域内で最大限大きく見える
- プレビューとPNG書き出しの見た目を一致させる

## 16. 制作手順としての推奨フロー

実際の制作では、次の順で調整する。

1. 文字を決める
2. 2行に分けるか決める
3. フォントを選ぶ
4. ウェイトを決める
5. 塗り色を決める
6. ライト背景で埋もれるか見る
7. 必要な場合だけ内側線を足す
8. ダーク背景で外周が見えるか見る
9. 外側線を調整する
10. 行間・字間・横幅を調整する
11. Slack/Discord風プレビューで確認する
12. デザインスコアと改善コメントで弱点を確認する

## 17. よくある失敗と修正

| 問題 | 原因 | 修正 |
|---|---|---|
| 黄色が白背景で読めない | 塗りと背景の分離不足 | 黒内側線を細く足す |
| 青文字が黒く詰まる | 不要な黒内側線が強い | 内側線を弱める/消す |
| ダーク背景で輪郭が消える | 外側線が弱い | 白外側線を 8〜14 程度にする |
| 漢字が潰れる | 高画数 + 太ウェイト + 線 | ウェイトか内側線を下げる |
| 絵文字が小さく見える | 透明余白・縦横比偏り | 幅揃え・横幅・トリミングを確認 |
| 文字が横に潰れる | 横幅補正が強すぎる | 85〜115 付近に戻す |
| 2行がぶつかる | 行間が狭い/線が太い | 行間を広げるか線幅を下げる |

## 18. 論文で主張できること

現時点で主張してよいこと:

```text
EmojiLab. は、カスタム絵文字制作において、色・縁取り・文字構成・小サイズ表示を同時に確認できる制作支援環境である。
```

```text
本システムは、APCAに基づくコントラスト評価と、文字数・画数・線幅・ウェイト・縦横比に基づく縮小耐性評価を統合し、制作中に即時フィードバックを提示する。
```

```text
黒白最大コントラストのみを最適解とせず、色の印象を保ちながら局所的な縁取りで視認性を補助する設計を採用している。
```

まだ検証が必要なこと:

- スコアと人間の読みやすさ評価がどの程度対応するか
- 改善コメントが実際の制作行動を変えるか
- 制作支援あり条件の成果物が第三者評価で高くなるか
- 色の印象を保つことが使用意欲や意味理解に寄与するか
