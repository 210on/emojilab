## 2. システムの研究的位置づけ

本システムは、チャットアプリ向け文字絵文字の**制作支援**と**視認性フィードバック**を同一画面で行う研究用プロトタイプである。目的は、専門的なデザイン訓練を受けていないユーザでも、

- 文字構成
- 縁取り構成
- 配色
- 小サイズ表示時の見え方

を反復的に調整できる環境を提供し、即時フィードバックによってデザイン品質を改善しやすくすることである。

重要なのは、本システムのスコアは**完成作品を絶対評価する審級指標**ではなく、**改善方向を提示する設計支援指標**である点である。

## 3. 生成ロジック

### 3.1 入力パラメータ

絵文字生成は以下の主要パラメータに依存する。

- 上段文字列 `textTop`
- 下段文字列 `textBottom`
- 上下サイズ比 `lineSizeBalance`
- フォント `fontFamily`
- フォントウェイト `fontWeight`
- 横方向補正 `condense`
- 字間 `letterSpacing`
- 行間 `spacing`
- 塗り色 `mainColor`
- 内側線の有無・色・幅 `stroke1*`
- 外側線の有無・色・幅 `stroke2*`
- 幅揃え `autoSquare`

### 3.2 上下サイズ比

上下サイズ比は、上段と下段に別々の倍率を与える。

- 正規化後の値は `-50` から `50`
- 上段倍率と下段倍率は反対方向に変化する
- 実装上の倍率は `1 ± 0.00857 × balance`
- 最終倍率は `0.56` から `1.44` にクランプされる

この処理により、2行構成の見た目の均衡を調整できる。

### 3.3 横幅補正と幅揃え

1. 各行の自然幅を canvas 上で実測する
2. `autoSquare` が無効なら、各行の幅はそのまま用いる
3. `autoSquare` が有効なら、2行の自然幅から共通目標幅を求める
4. その後、ユーザ指定の横方向補正 `condense` を掛ける

`autoSquare` の目標幅は、単に長い行へ合わせるのではなく、2行の自然幅から求めた共通幅へ両方を近づける。実装上は幾何平均ベースで決めており、片方だけを極端に引き延ばしたり潰したりしにくい。

### 3.4 行間

行間は次の式で算出する。

- `desiredGap = size * 0.082 + spacing * (size / 700)`
- その後 `0` から `size * 0.22` にクランプ

ここでの `spacing` は UI 上の行間操作量であり、最終的にはキャンバスサイズに比例してスケーリングされる。

### 3.5 線の描画順

本システムは線を単純な `strokeText()` で描かず、マスク画像の膨張処理によって帯状の線レイヤーを構成する。

処理順:

1. 塗り文字のマスクを生成
2. 内側線をマスク膨張で生成
3. 外側線をさらにその外側に生成
4. 合成順は `外側線 → 内側線 → 塗り`

この構成により、

- 横方向の伸縮時にも線幅の見え方が破綻しにくい
- 塗りが線で覆われにくい
- PNG 書き出しとプレビューの描画差を減らせる

### 3.6 余白除去と正方形化

描画後の画像は、透明画素を走査して不透明境界を検出し、

- 透明余白を除去した `trimmedCanvas`
- 長辺基準で正方形化した `squareCanvas`

を生成する。PNG 書き出しは前者を用いる。チャットプレビューやガイド表示は後者を利用する。

## 4. デザインスコアの構成

### 4.1 表示される3指標

UI 上の `デザインスコア` では、次の3値を扱う。

- `contrastRatio`
- `scalability`
- `overallScore`

このうち、円形メーター中央の数値は `overallScore` である。

### 4.2 contrastRatio

`contrastRatio` は、塗り色・内側線・外側線・背景色の組合せから算出する**APCA-W3 参照実装ベースの知覚コントラスト指標**である。UIでは、内部の `displayedContrastLc` を `Lc` として表示する。

現行実装では、次の2種類を分けて評価する。

- `localTextLc`: 塗り色と、有効な内側線または外側線の局所コントラスト
- `backgroundSeparationLc`: 塗り・有効な内側線・有効な外側線のうち、各背景に対して最も分離する層のコントラスト

表示値は、上記2値の小さい方を用いる。

実装には `apca-w3` の `APCAcontrast()` と `sRGBtoY()` を用いる。APCA は極性依存の signed Lc を返すため、本システムでは UI 表示と閾値判定の一貫性を保つために絶対値へ正規化して扱う。すなわち、内部では APCA の参照実装を用い、表示上は「見え方の差の大きさ」を単一スカラーとして提示している。

有効線の初期判定は、内側線 `2` 以上、外側線 `4` 以上とする。外側線の安定域は `8-14` とし、細すぎる線は APCA 値が高くても境界として過大評価しない。

高明度・高彩度の塗り色が白背景で弱い場合は、塗り色を暗くする提案よりも、黒など十分に差のある内側線で局所コントラストを補う提案を優先する。ただし、白背景で十分に読める色では、太い内側線は縮小耐性側で複雑性リスクとして扱う。

### 4.3 scalability

`scalability` は、小サイズ表示時の保持性を近似するヒューリスティック指標である。初期値 100 から、以下のように加減点する。

- 文字数が多い: 減点
- フォントウェイトが細い: 減点
- フォントウェイトが太い: 低画数では軽微に加点し、高画数漢字では詰まりリスクとして減点
- 内側線が太すぎる: 減点
- 外側線が推奨幅 `8-14` に入る: 加点
- 外側線が `2-7` と細い: 減点
- 外側線が `15` 以上: 推奨幅より弱い扱い
- 外側線が無い: 減点

現行実装では、文字数、画数、フォントウェイト、線幅、横幅補正を同時に見て減点する。特に、高画数漢字と太ウェイト、強い内側線、強い圧縮の組合せは小サイズで詰まりやすいため、相互作用として追加減点する。

### 4.4 overallScore

`overallScore` は、コントラスト、縮小耐性、文字構成を統合した**ルールベースの総合支援スコア**である。式は以下である。

- `0.45 × contrastFitScore`
- `0.35 × scalabilityScore`
- `0.20 × compositionStabilityScore`

ここで `compositionStabilityScore` は、以下の要素で構成される。

- 行間
- 字間
- 横幅補正
- 上下サイズ比
- 上下行の幅バランス
- 推定外接矩形の縦横比

画数評価は `scalabilityScore` に含める。評価は [kanjiStrokeCounts.ts](/Users/shonebato/Documents/New%20project/custom-emoji-studio/utils/kanjiStrokeCounts.ts) の画数辞書に基づく。現行実装では、Unicode Unihan の `kTotalStrokes` を出典とし、`kJoyoKanji`、`kJinmeiyoKanji`、`kJis0`、`kJis1`、`kJIS0213` に該当する漢字と、実用上補完した日本語異体字 `髙` を収録している。登録数は `13,110` 字である。

この辞書は、常用漢字・人名用漢字・JIS系漢字を広く含むため、以前の手動軽量辞書より未登録漢字を大幅に減らしている。ただし、すべてのHan文字を網羅するものではない。JIS/Joyo/Jinmeiyo範囲外の拡張漢字や一部異体字では `unknownKanjiCount` が発生し得る。

画数指標は次の値として扱う。

- `totalStrokeCount`: 画数辞書に登録された漢字の総画数
- `maxStrokeCount`: 1文字あたりの最大画数
- `denseKanjiCount`: 14画以上の既知漢字数
- `unknownKanjiCount`: CJK漢字だが画数辞書に未登録の文字数

主な減点条件は次の通りである。

- 最大画数が14画以上: 小サイズでの字面密度リスクとして減点
- 最大画数が16画以上: 高画数漢字として強めに減点
- 最大画数が20画以上: かなり高密度な文字として強く減点
- 総画数が24画以上または36画以上: 文字列全体の密度として減点
- 高画数漢字と `fontWeight >= 800` の併用: 太ウェイトによる詰まりとして追加減点
- 高画数漢字と強い内側線の併用: 字面内部の詰まりとして追加減点
- 未登録漢字を含む場合: JIS/Joyo/Jinmeiyo範囲外の未知リスクとして軽く減点

### 4.5 良好 / 要改善の閾値

表示ラベルは次の条件で決まる。

- `excellent`
  - `contrastRatio >= 75`
  - `scalability >= 82`
  - `overallScore >= 84`
- `good`
  - `contrastRatio >= 60`
  - `scalability >= 72`
  - `overallScore >= 70`
- それ以外
  - `needsWork`

この設計により、単一要因ではなく、コントラスト・縮小耐性・構成評価の複合状態として「良好」を判定する。

改善コメント内部の `acceptable` 判定も、UI 上の `good` 判定と同様に `overallScore >= 70` を要求する。これにより、コントラストと縮小耐性だけが許容域でも、総合支援スコアが低い場合に肯定的な文言が出ないようにしている。

### 4.6 改善コメント

下部コメント `tip` は、最も改善インパクトが大きい要因を1文で返すよう設計されている。

現行実装では、固定の優先順位だけで文言を決めない。まず、コントラスト不足、背景分離不足、高画数漢字と太ウェイト、内側線過多、外側線不足、縦横比の偏りなどの改善候補を列挙し、それぞれに推定インパクトを与える。そのうえで、最も総合支援スコアの上昇幅が大きいと見込まれる候補を提示する。

推定インパクトは、以下のような観点で近似する。

- コントラスト適合の不足量
- 縮小耐性の不足量
- その問題が現行スコア式で直接減点されているか
- 線幅や画数など、ユーザーが一手で修正しやすいパラメータか
- すでに許容域に入っている場合は、強い改善指示ではなく軽い調整または肯定コメントにする

したがって、このコメントは総評ではなく、**次に操作すべきパラメータを示す設計誘導**である。

## 5. 学術的根拠

### 5.1 コントラストを重視する根拠

コントラストが読みやすさに強く影響すること自体は、アクセシビリティ研究と視覚可読性研究の基本前提である。W3C の視覚コントラスト議論でも、読みやすさには単なる識別可能性より高いコントラストが必要であり、サイズや線の細さと相互依存すると整理されている。

参考:

- W3C Silver wiki, *Visual Contrast of Text Subgroup*  
  <https://www.w3.org/WAI/GL/task-forces/silver/wiki/Visual_Contrast_of_Text_Subgroup>
- W3C, *WCAG 2.2*  
  <https://www.w3.org/TR/WCAG22/>

### 5.2 サイズ・線幅・文字密度を重視する根拠

小さい文字ほど、視力限界だけでなくコントラスト感度の低下や高空間周波数への脆弱性を受けやすい。W3C の同資料でも、細い文字・小さい文字は低コントラストに見えやすいこと、サイズとウェイトが可読性と相互依存であることが明示されている。

現行実装の `scalability` は、この知見を研究用 UI に持ち込むための操作可能な近似指標である。

### 5.3 文字複雑性を重視する根拠

漢字や記号を含む文字列では、画数や構成の複雑さが小サイズ時の判別負荷を増やす。現行システムではこれを、字数・画数・記号・線幅ヒューリスティックで近似している。

ただし、これは現時点では**実験的な構成評価**であり、独立した標準尺度ではない。画数評価は Unicode Unihan 由来の再現可能なデータに基づくが、文字の見えやすさを直接測定するものではないため、論文では「文字構成の複雑さを補助的に評価するための設計支援ヒューリスティック」と記述するのが適切である。

## 6. 論文で主張してよいこと

以下は比較的強く主張できる。

- 本システムは、コントラスト、縮小耐性、文字構成の観点から文字絵文字を即時評価する制作支援環境である。
- 外側線、内側線、字間、行間、横幅、上下サイズ比などの可変パラメータを持ち、評価結果に基づいて反復調整できる。
- デザインスコアは、視認性に関わる複数要因を単一 UI に統合した設計支援指標である。

## 7. 論文で慎重に書くべきこと

以下は断定しないほうがよい。

- 「本スコアは人間の視認性を客観的・厳密に測定する」
- 「本スコアは美的品質や好ましさまで評価する」

現行実装は、研究用プロトタイプとしては十分に筋が良いが、心理物理実験で校正された単一尺度ではない。

## 8. 論文向けの推奨表現

### 8.1 短い定義

> 本システムのデザインスコアは、コントラスト推定、縮小耐性ヒューリスティック、および文字構成評価に基づく設計支援指標である。

### 8.2 より安全な定義

> 本スコアは完成作品を絶対評価するための指標ではなく、配色・縁取り・文字選択の改善方向を即時に提示する相対的なフィードバック指標として設計した。

## 9. 妥当性をさらに強めるための今後の実験

論文としての妥当性をさらに強めるなら、以下が有効である。

1. 被験者に複数案を提示し、主観評価と `contrastRatio / scalability / overallScore` の相関を見る
2. 画数・線幅・ウェイト条件を分けて評価する
3. ライト背景・ダーク背景・チャットプレビューでの一貫性を比較する
4. 文字種別
   - ひらがな
   - カタカナ
   - 低画数漢字
   - 高画数漢字
   - 記号混在
   でスコア挙動を検証する

## 10. 要約

現行の Custom Emoji Studio は、厳密な意味での単一客観尺度ではなく、

- 知覚コントラスト
- 小サイズ表示時の保持性
- 文字構成の複雑さ

を統合した**研究用の設計支援システム**である。したがって論文では、「デザインスコア」を**視認性評価を支援する複合指標**として位置づけるのが最も妥当である。
