# Custom Emoji Studio デザイン設計指針

## 1. 文書の位置づけ

本書は、Custom Emoji Studio を研究用途で運用する際に、

- どの設計条件を「推奨値」とみなすか
- どこまで既存研究に基づいて主張できるか
- どこから先が本研究の仮説であるか

を整理するための文書である。

重要なのは、**チャット向けカスタム文字絵文字そのものに特化した確立研究は多くない**という点である。したがって本書の推奨は、

1. 視覚可読性・タイポグラフィ研究
2. W3C / APCA 系の視覚コントラスト研究
3. 現行システム実装の操作変数

を接続した**研究上の設計ガイドライン**として扱う。

## 2. APCA の厳密実装は可能か

可能である。

現状の実装 [App.tsx](/Users/shonebato/Documents/New%20project/custom-emoji-studio/App.tsx) では、`apca-w3` による厳密な参照実装を用いている。

APCA の公式側ドキュメントでは、開発用コードとして `apca-w3` を正規コードとして扱っている。

参考:

- APCA Documentation Index  
  [https://git.apcacontrast.com/documentation/](https://git.apcacontrast.com/documentation/)
- Myndex/SAPC-APCA  
  [https://github.com/Myndex/SAPC-APCA](https://github.com/Myndex/SAPC-APCA)

上記では、`apca-w3` が「development purpose に使う approved code」であると明示されている。

### 2.1 研究上の推奨

論文で `APCA` を強く前面に出すなら、`apca-w3` の参照実装を使うのが望ましい。本システムはこの条件を満たす。

推奨理由:

- APCA を名乗る際の実装妥当性が上がる
- 閾値 `Lc 45 / 60 / 75 / 90` を公式系資料に結びつけやすい
- 論文査読で「独自近似ではないか」という指摘を受けにくい

### 2.2 実装方針

研究用途としては以下が妥当である。

1. `apca-w3` で前景色・背景色から `Lc` を算出
2. UI では signed Lc の絶対値を用いて閾値判定する
3. 評価対象色は、文字塗りと隣接する線色または背景色の組合せから選ぶ
4. 閾値の意味は「極性込みの APCA 出力」ではなく「知覚差の大きさ」として解釈する

### 2.3 論文での書き方

APCA を厳密実装へ置き換えた後は、以下のように書ける。

> コントラスト評価には、Myndex が公開する APCA-W3 の参照実装を用いた。

本システムのように UI 側で絶対値を表示する場合は、次のように補うと誤解が少ない。

> コントラスト評価には、Myndex が公開する APCA-W3 の参照実装を用いた。なお、UI 上の表示値は極性依存の signed Lc を絶対値化したものである。

## 3. 研究に基づく設計指針

### 3.0 推奨条件一覧

| 観点 | 推奨条件 | 許容条件 | 非推奨条件 | 根拠の強さ |
| --- | --- | --- | --- | --- |
| コントラスト | `Lc >= 75` | `Lc >= 60` | `Lc < 45` | 高 |
| フォント | ゴシック / 丸ゴシック | 明朝を比較条件として使用 | 装飾的 display font を標準条件にする | 中 |
| ウェイト | `700-900` | `600-700` | `400` 未満 | 中 |
| 外側線 | 常時有効、主輪郭として使用 | 太さを背景に応じて調整 | 外側線なし | 中 |
| 内側線 | `3-6` 程度で補助的に使用 | フォントや字数に応じて弱める | 太すぎて塗り面を侵食する | 中 |
| 横幅補正 | `90-110` | `85-115` | `72` 未満または `128` 超 | 中 |
| 文字数 | `2-4` 文字 | `5` 文字前後は比較条件 | 長文・記号過多 | 中 |
| 文字種 | かな / 低画数漢字 | 高画数漢字を比較条件にする | 高画数 + 太ウェイト + 強い内側線の併用 | 低-中 |

ここでいう「根拠の強さ」は、APCA や W3C で比較的直接支えられるものを「高」、一般的な可読性研究と実装上の一貫した知見から支えるものを「中」、本システムの仮説的運用指針に近いものを「低-中」として整理したものである。

## 3.1 色選択

### 根拠

W3C Silver の視覚コントラスト議論では、可読性は単純な識別可能性より高いコントラストを必要とし、文字サイズや線の細さと相互依存すると整理されている。

参考:

- W3C Silver: Visual Contrast of Text Subgroup  
  [https://www.w3.org/WAI/GL/task-forces/silver/wiki/Visual_Contrast_of_Text_Subgroup](https://www.w3.org/WAI/GL/task-forces/silver/wiki/Visual_Contrast_of_Text_Subgroup)
- WCAG 2.2  
  [https://www.w3.org/TR/WCAG22/](https://www.w3.org/TR/WCAG22/)

### 推奨

1. 塗り色単体の「好きな色」より、**塗り色と外側線色の明度差**を優先する
2. 研究用の「許容域」は `Lc >= 60`
3. 良好域は `Lc >= 75`
4. 小サイズ主体の主要候補は `Lc >= 75` を優先
5. `Lc < 45` は原則として改善対象とする

### 色相ごとの扱い

- 黄・橙・赤系  
  高彩度でも注意を引きやすいが、明度が高い場合は外側線が弱いと埋もれやすい
- 青・紫・水色系  
  印象は保ちやすいが、暗背景・低輝度環境では見えが不安定になりやすい
- 白系・淡色系  
  塗りだけでは不安定になりやすいため、強い外側線を前提に使う
- 灰色系  
  中庸だが、背景との差が不足しやすいのでコントラスト閾値で厳格にふるい分ける

### 研究上の運用推奨

- ライト / ダーク両背景で確認する
- 単色の美しさではなく、**背景を含む局所コントラスト**で採否を決める
- 提示する推奨色群は、PCCS のような体系化された色群から選ぶと再現性が高い

## 3.2 枠線スタイル

### 根拠

小さく細い文字は、コントラスト感度の低下や高空間周波数への脆弱性の影響を受けやすい。W3C の議論でも、フォントサイズとウェイトは可読性と相互依存であるとされる。

そのため、文字絵文字では塗り単体よりも、**外側線による輪郭保持**が重要になる。

### 推奨

1. 外側線を主輪郭、内側線を補助輪郭として扱う
2. 外側線は原則有効
3. 外側線は内側線より重要
4. 内側線は強すぎると字面を潰すため、太字フォントとの併用では抑制する

### 運用ルール

- 最低条件  
  外側線あり
- 安定条件  
  外側線が十分視認できる太さ
- 避けたい条件  
  内側線が太く、塗り面積を侵食する状態

### 現行システムに即した推奨

- 内側線: `3–6`
- 外側線: `8–14`
- 外側線なし: 原則非推奨

これは厳密な普遍値ではなく、現行キャンバススケールと UI スケールに対応した運用値である。

## 3.3 フォント選択

### 根拠

可読性研究では、字形の基本骨格、カウンターの大きさ、ウェイト、ストロークの単純さが legibility に影響する。装飾過多や極端な condensed / expanded は一般に不利である。

参考:

- Readability Research: An Interdisciplinary Approach  
  [https://arxiv.org/abs/2107.09615](https://arxiv.org/abs/2107.09615)

### 推奨

1. ゴシック体・丸ゴシック体を主軸にする
2. 明朝体は比較対象としては有用だが、主要候補としては慎重に扱う
3. 極端に装飾的な display font は小サイズ用途では避ける
4. 文字骨格が明快で、閉じたカウンターが潰れにくいものを優先する

### 研究用の優先順位

- 第一候補  
  ゴシック
- 第二候補  
  丸ゴシック
- 第三候補  
  明朝系
- 特殊比較  
  装飾書体・筆文字風

### 論文での解釈

フォント比較を行う場合は、「美的印象」ではなく、

- 骨格の単純さ
- 閉じ部の保持性
- 小サイズでの輪郭維持

という観点で比較軸を置くのが妥当である。

## 3.4 フォントウェイト

### 根拠

細い文字は低コントラストに見えやすく、小サイズでは不利になる。反対に、太すぎると内側線や複雑文字との組合せで潰れやすい。

### 推奨

- 主要推奨帯: `700–900`
- 研究上の標準条件: `800` または `900`
- `400` 未満: 非推奨
- 複雑な漢字を含む場合: `800–900` を優先

### 実務的な判断

- ひらがな主体  
  `700–800` でも可
- 漢字主体  
  `800–900` を推奨
- 外側線が弱い場合  
  ウェイトを上げるより外側線を強めるほうが再現性が高い

## 3.5 縦横比・横幅圧縮

### 根拠

可読性研究では、基本骨格から大きく逸脱した condensed / expanded 文字は legibility を損ねやすい。したがって、横幅補正は可能でも、研究条件としては自然幅付近に制限したほうがよい。

### 推奨

- 自然域: `90–110`
- 実験許容域: `85–115`
- 強い補正を許す上限: `80–120`
- `72` 未満または `128` 超は、比較実験条件としては非推奨

### 現行システムとの関係

現行 UI の可変域はより広いが、これは探索のための操作域であり、論文で「推奨条件」として採用する範囲は絞るべきである。

## 3.6 文字選択

### 根拠

高画数文字や記号混在は、小サイズ表示で判別負荷を増やす。現行システムの AI / ルールベース評価もこの前提を反映している。

ルールベース評価では、軽量な漢字画数辞書を用いて、既知漢字の総画数、最大画数、14画以上の高画数漢字数、辞書未登録漢字数を計測している。これは人間の視認性を直接測定するものではなく、小サイズ表示で字面が詰まりやすい条件を検出するための補助指標である。

### 推奨

- 2〜4文字程度を主条件とする
- ひらがな・カタカナ・低画数漢字を優先
- 高画数漢字は比較対象にはなるが、標準解にはしない
- 記号は 1 つ程度までを基本とする
- 14画以上の漢字を含む場合は、ウェイトや内側線を控えめにする
- 16画以上の漢字を太ウェイトで使う場合は、チャットプレビュー上で潰れを重点確認する

### 画数評価の運用

- `maxStrokeCount >= 14`  
  高画数文字として注意対象にする
- `maxStrokeCount >= 16`  
  太ウェイトや強い内側線との併用を避ける
- `maxStrokeCount >= 20`  
  小サイズでは標準条件にしない
- `totalStrokeCount >= 24`  
  文字列全体の密度が高い条件として扱う
- `unknownKanjiCount > 0`  
  画数辞書に未登録の漢字を含むため、保守的に評価する

## 4. 論文に書ける推奨仕様

以下は、現行研究段階で比較的 defensible な推奨仕様である。

- コントラスト  
  `Lc >= 60` を最低条件、`Lc >= 75` を推奨条件
- 縮小耐性  
  `72` 以上を許容、`82` 以上を良好
- フォント  
  ゴシックまたは丸ゴシック
- ウェイト  
  `700–900`。ただし高画数漢字では `700–800` を優先し、`900` は潰れ確認を前提にする
- 内側線  
  `3–6`
- 外側線  
  `8–14`
- 横幅補正  
  `85–115` を主条件

## 5. 論文に書くときの安全な言い方

### 強く言ってよい

> 本研究では、小サイズ表示時の視認性に関わる主要因として、知覚コントラスト、文字の輪郭保持、字形の複雑さ、ならびにフォントウェイトと横幅補正を扱った。

### 慎重に言うべき

> 本研究で提示する推奨値は、既存の可読性研究と APCA 系コントラスト研究を参照しつつ、文字絵文字の制作条件に適用可能な形で構成した設計指針である。

### 断定しないほうがよい

- 「この値が普遍的最適である」
- 「この色が常に最良である」
- 「この枠線幅が全フォント・全背景で最適である」

## 6. 現時点での研究的結論

現時点で最も学術的に筋が良い結論は次である。

1. APCA の厳密実装は可能であり、本システムでは `apca-w3` を用いている
2. カスタム文字絵文字の設計では、色相そのものよりも明度差と外側線の安定性が重要である
3. フォントはゴシック系、ウェイトは `700–900`、横幅補正は自然幅近傍が妥当である
4. 外側線を基軸とし、内側線は補助として用いる構成が、小サイズ視認性の観点で合理的である
5. これらは厳密な「最適解」ではなく、視認性を保ちながら表現の自由度も残すための研究上の推奨条件である
