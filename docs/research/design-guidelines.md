# EmojiLab. デザイン設計指針

## 1. 文書の位置づけ

本書は、EmojiLab. を研究用途で運用する際に、

- どの設計条件を「推奨値」とみなすか
- どこまで既存研究に基づいて主張できるか
- どこから先が本研究の仮説であるか

を整理するための文書である。

重要なのは、**チャット向けカスタム文字絵文字そのものに特化した確立研究は多くない**という点である。したがって本書の推奨は、

1. 視覚可読性・タイポグラフィ研究
2. W3C / APCA 系の視覚コントラスト研究
3. 現行システム実装の操作変数

を接続した**研究上の設計ガイドライン**として扱う。

## 2. APCA の厳密実装は可能か

可能である。

現状の実装 [App.tsx](/Users/shonebato/Documents/New%20project/custom-emoji-studio/App.tsx) では、`apca-w3` による厳密な参照実装を用いている。

APCA の公式側ドキュメントでは、開発用コードとして `apca-w3` を正規コードとして扱っている。

参考:

- APCA Documentation Index  
  [https://git.apcacontrast.com/documentation/](https://git.apcacontrast.com/documentation/)
- Myndex/SAPC-APCA  
  [https://github.com/Myndex/SAPC-APCA](https://github.com/Myndex/SAPC-APCA)

上記では、`apca-w3` が「development purpose に使う approved code」であると明示されている。

### 2.1 研究上の推奨

論文で `APCA` を強く前面に出すなら、`apca-w3` の参照実装を使うのが望ましい。本システムはこの条件を満たす。

推奨理由:

- APCA を名乗る際の実装妥当性が上がる
- 閾値 `Lc 45 / 60 / 75 / 90` を公式系資料に結びつけやすい
- 論文査読で「独自近似ではないか」という指摘を受けにくい

### 2.2 実装方針

研究用途としては以下が妥当である。

1. `apca-w3` で前景色・背景色から `Lc` を算出
2. UI では signed Lc の絶対値を用いて閾値判定する
3. 評価対象色は、文字塗りと隣接する線色または背景色の組合せから選ぶ
4. 閾値の意味は「極性込みの APCA 出力」ではなく「知覚差の大きさ」として解釈する

### 2.3 論文での書き方

APCA を厳密実装へ置き換えた後は、以下のように書ける。

> コントラスト評価には、Myndex が公開する APCA-W3 の参照実装を用いた。

本システムのように UI 側で絶対値を表示する場合は、次のように補うと誤解が少ない。

> コントラスト評価には、Myndex が公開する APCA-W3 の参照実装を用いた。なお、UI 上の表示値は極性依存の signed Lc を絶対値化したものである。

## 3. 研究に基づく設計指針

### 3.0 推奨条件一覧

| 観点 | 推奨条件 | 許容条件 | 非推奨条件 | 根拠の強さ |
| --- | --- | --- | --- | --- |
| コントラスト | `Lc >= 75` | `Lc >= 60` | `Lc < 45` | 高 |
| フォント | ゴシック / 丸ゴシック | 明朝を比較条件として使用 | 装飾的 display font を標準条件にする | 中 |
| ウェイト | 標準条件 `700` | 高画数では `600-800` も比較 | `400` 未満、または高画数 + `900` | 低-中 |
| 外側線 | 背景分離が不足する場合に使用 | 初期安定域 `8-14` | 細線を十分な境界として扱う | 低-中 |
| 内側線 | 色差補助が必要な場合に細く使用 | フォントや字数に応じて弱める | 太すぎて塗り面を侵食する | 低-中 |
| 横幅補正 | 初期無減点域 `90-110` | `85-115` を検証域にする | 強い変形を未検証のまま推奨する | 低 |
| 文字数 | `2-4` 文字 | `5` 文字前後は比較条件 | 長文・記号過多 | 中 |
| 文字種 | かな / 低画数漢字 | 高画数漢字を比較条件にする | 高画数 + 太ウェイト + 強い内側線の併用 | 低-中 |

ここでいう「根拠の強さ」は、APCA や W3C で比較的直接支えられるものを「高」、一般的な可読性研究と実装上の一貫した知見から支えるものを「中」、本システムの仮説的運用指針に近いものを「低-中」として整理したものである。

## 3.1 色選択

### 根拠

W3C Silver の視覚コントラスト議論では、可読性は単純な識別可能性より高いコントラストを必要とし、文字サイズや線の細さと相互依存すると整理されている。

参考:

- W3C Silver: Visual Contrast of Text Subgroup  
  [https://www.w3.org/WAI/GL/task-forces/silver/wiki/Visual_Contrast_of_Text_Subgroup](https://www.w3.org/WAI/GL/task-forces/silver/wiki/Visual_Contrast_of_Text_Subgroup)
- WCAG 2.2  
  [https://www.w3.org/TR/WCAG22/](https://www.w3.org/TR/WCAG22/)

### 推奨

1. 塗り色単体の「好きな色」より、**塗り色と外側線色の明度差**を優先する
2. 研究用の「許容域」は `Lc >= 60`
3. 良好域は `Lc >= 75`
4. 小サイズ主体の主要候補は `Lc >= 75` を優先
5. `Lc < 45` は原則として改善対象とする

### 色相ごとの扱い

- 黄・橙・赤系  
  高彩度でも注意を引きやすいが、明度が高い場合は外側線が弱いと埋もれやすい
- 青・紫・水色系  
  印象は保ちやすいが、暗背景・低輝度環境では見えが不安定になりやすい
- 白系・淡色系  
  塗りだけでは不安定になりやすいため、強い外側線を前提に使う
- 灰色系  
  中庸だが、背景との差が不足しやすいのでコントラスト閾値で厳格にふるい分ける

### コントラスト不足色に対する黒内側線

黒内側線の推奨は、PCCS番号や色名を固定条件にするのではなく、背景とのコントラスト値と線の有効幅にもとづいて決める。

基本ルール:

- 白背景との APCA が低い
- 塗り色がOKLCH上で高明度・高彩度に分類され、暗色化すると色の印象が変わりやすい
- 内側線が小サイズでも視認できる幅を持つ

この条件を満たす場合は、色そのものを暗くするのではなく、黒い内側線や外側線によって局所コントラストを補う方針が合理的である。PCCS bright tone の現行12色では、このルールを適用した結果として `b6`（橙）、`b8`（黄）、`b10`（黄緑）が主な該当例になる。

一方で、白背景に対して十分なコントラストを持つ色では、黒い内側線を強く入れると小サイズで字面が詰まりやすい。したがって、黒内側線は全色に一律適用するのではなく、コントラスト不足と小サイズでの線の有効性に応じて適用する。

OKLCH上の初期定義:

- 高明度: `L >= 0.70`
- 高彩度: `C >= 0.10`

この値は絶対的な知覚基準ではなく、色特徴量を再現可能に扱うための操作的定義である。最終的な妥当性は、Study 1 / Study 2 の人間評価との対応によって検証する。

### 研究上の運用推奨

- ライト / ダーク両背景で確認する
- 単色の美しさではなく、**背景を含む局所コントラスト**で採否を決める
- 提示する推奨色群は、PCCS のような体系化された色群から選ぶと再現性が高い

## 3.2 枠線スタイル

### 根拠

小さく細い文字は、コントラスト感度の低下や高空間周波数への脆弱性の影響を受けやすい。W3C の議論でも、フォントサイズとウェイトは可読性と相互依存であるとされる。

そのため、文字絵文字では塗り、隣接する線、背景の境界を一体として確認する必要がある。外側線は背景分離を補えるが、十分な色差がすでにある場合まで必須とはいえない。

### 推奨

1. 外側線は背景分離が不足する場合の補助手段として扱う
2. 内側線は明るい塗りとライト背景の間に局所色差が必要な場合に使う
3. 内側線は強すぎると字面を潰すため、太字フォントとの併用では抑制する
4. 線が細すぎる場合は、APCA値が高くても実効的な境界として過大評価しない

### 運用ルール

- 最低条件  
  塗りまたは有効線が各対象背景から十分に分離する
- 安定条件  
  外側線が十分視認できる太さ
- 避けたい条件  
  内側線が太く、塗り面積を侵食する状態
- 注意条件  
  内側線・外側線が細すぎ、小サイズ表示でアンチエイリアスに埋もれる状態

### 現行システムに即した推奨

- 内側線: `3–6`
- 外側線: `8–14`
- 外側線なし: 背景分離が十分なら線幅減点なし

これは厳密な普遍値ではなく、現行キャンバススケールと UI スケールに対応した運用値である。

### 有効線の考え方

枠線は、単にON/OFFだけで評価しない。線が細すぎる場合、数理上は高いコントラストを持っていても、16px〜20px表示では視覚的な境界として十分に機能しない可能性がある。

研究用スコアでは、以下のように線の有効性を扱う。

- 内側線 `2` 未満  
  原則として補助線としては弱い
- 外側線 `4` 未満  
  背景分離としては弱い
- 外側線 `8–14`  
  主輪郭として安定しやすい
- 内側線 `8` 以上  
  高画数漢字や太ウェイトでは潰れリスクが高い

これは、線の色差だけでなく、線が実際に見えるだけの幅を持っているかを評価に含めるためである。

現行実装では、まずUI上の線幅値による有効線判定を採用している。16px / 20px / 32px表示に換算した実効線幅での判定は、Slack / Discord 上の実測表示サイズを研究データとして固定した後に導入する。特に、16px表示で1px未満になる線は境界として弱く、2px以上相当になる内側線は高画数文字で潰れリスクを持つ、という仮説を次段の検証対象とする。

## 3.3 フォント選択

### 根拠

可読性研究では、字形の基本骨格、カウンターの大きさ、ウェイト、ストロークの単純さが字形判別性に影響する。装飾過多や極端な condensed / expanded は一般に不利である。

参考:

- Readability Research: An Interdisciplinary Approach  
  [https://arxiv.org/abs/2107.09615](https://arxiv.org/abs/2107.09615)

### 推奨

1. ゴシック体・丸ゴシック体を主軸にする
2. 明朝体は比較対象としては有用だが、主要候補としては慎重に扱う
3. 極端に装飾的な display font は小サイズ用途では避ける
4. 文字骨格が明快で、閉じたカウンターが潰れにくいものを優先する

### 研究用の優先順位

- 第一候補  
  ゴシック
- 第二候補  
  丸ゴシック
- 第三候補  
  明朝系
- 特殊比較  
  装飾書体・筆文字風

### 論文での解釈

フォント比較を行う場合は、「美的印象」ではなく、

- 骨格の単純さ
- 閉じ部の保持性
- 小サイズでの輪郭維持

という観点で比較軸を置くのが妥当である。

## 3.4 フォントウェイト

### 根拠

細い文字は低コントラストに見えやすく、小サイズでは不利になる。反対に、太すぎると内側線や複雑文字との組合せで潰れやすい。

### 初期運用条件

- 研究上の標準条件: `700`
- `400` 未満: 細線消失リスクとして減点
- `400–599`: 軽い減点
- 高画数漢字と `900` 以上: 詰まりの相互作用として減点

これらはフォントごとの実画線幅を直接測った閾値ではない。細すぎ・太すぎの両方向を比較できる初期条件であり、実寸認識実験で校正する。

### 実務的な判断

- ひらがな主体  
  `700` を基準に比較する
- 漢字主体  
  `700` を基準とし、高画数では `600–800` を比較する
- 外側線が弱い場合  
  ウェイトを上げるより外側線を強めるほうが再現性が高い

## 3.5 縦横比・横幅圧縮

### 根拠

Minakata and Beier (2021) はフォント幅によって注視時間やサッカードが変化し得ることを示した一方、読了時間と理解正答には有意な主効果を報告していない。このため、横幅は評価対象に含める根拠はあるが、特定の圧縮率が直ちに認識を損なうとは断定できない。

### 推奨

- 初期無減点域: `90–110`
- 初期検証域: `85–115`
- それを超える変形: 段階的な減点候補

上記の数値は文献から直接得た限界値ではなく、自然幅からの偏差を段階化する EmojiLab. の操作的定義である。

### 現行システムとの関係

現行 UI の可変域はより広いが、これは探索のための操作域であり、論文で「推奨条件」として採用する範囲は絞るべきである。

### 正方形プロポーション

Slack や Discord のカスタム絵文字は正方形領域で表示される。そのため、実際の不透明領域が極端に縦長または横長になると、表示領域を使い切れず、視認性上のロスが生じる。

ただし、正方形性は可読性や意味表現より優先されるべきではない。研究用スコアでは弱い補助指標として扱い、主に改善コメントで提示する。

運用値:

- `0.80 <= width / height <= 1.25`  
  安定
- `0.70–0.80` または `1.25–1.40`  
  軽い注意
- `0.70` 未満または `1.40` 超  
  構成上のロスとして扱う

改善コメント例:

> 全体が縦長になっているため、横幅を少し広げるか幅揃えを使うと、正方形の絵文字枠をより有効に使えます。

実装上は、塗り領域のみの `coreAspectRatio` と、線を含めた `fullAspectRatio` を分ける。縮小耐性では `coreAspectRatio` を重視し、書き出し領域の利用効率や改善コメントでは `fullAspectRatio` を参照する。

## 3.6 文字選択

### 根拠

高画数文字や記号混在は、小サイズ表示で判別負荷を増やす。現行システムのルールベース総合支援スコアもこの前提を反映している。

ルールベース評価では、Unicode Unihan の `kTotalStrokes` から生成した漢字画数辞書を用いて、既知漢字の総画数、最大画数、14画以上の高画数漢字数、辞書未登録漢字数を計測している。対象は `kJoyoKanji`、`kJinmeiyoKanji`、`kJis0`、`kJis1`、`kJIS0213` とし、実用上必要な日本語異体字 `髙` を補完している。登録数は `13,110` 字である。

これは人間の視認性を直接測定するものではなく、小サイズ表示で字面が詰まりやすい条件を検出するための補助指標である。

### 推奨

- 2〜4文字程度を主条件とする
- ひらがな・カタカナ・低画数漢字を優先
- 高画数漢字は比較対象にはなるが、標準解にはしない
- 記号は 1 つ程度までを基本とする
- 14画以上の漢字を含む場合は、ウェイトや内側線を控えめにする
- 16画以上の漢字を太ウェイトで使う場合は、チャットプレビュー上で潰れを重点確認する

### 画数評価の運用

- `maxStrokeCount >= 14`  
  高画数文字として注意対象にする
- `maxStrokeCount >= 16`  
  太ウェイトや強い内側線との併用を避ける
- `maxStrokeCount >= 20`  
  小サイズでは標準条件にしない
- `totalStrokeCount >= 24`  
  文字列全体の密度が高い条件として扱う
- `unknownKanjiCount > 0`  
  画数辞書に未登録の漢字を含むため、保守的に評価する

## 4. 論文に書ける推奨仕様

以下は、現行研究段階で比較的 defensible な推奨仕様である。

- コントラスト  
  `Lc >= 60` を最低条件、`Lc >= 75` を推奨条件
- 縮小耐性  
  `72` 以上を許容、`82` 以上を良好
- フォント  
  ゴシックまたは丸ゴシック
- ウェイト  
  標準条件を `700` とし、高画数漢字では `600–800` を比較、`900` は潰れ確認を前提にする
- 内側線  
  `3–6`。ただし、背景とのコントラスト補助が不要な色では太すぎる黒内側線を避ける
- 外側線  
  `8–14`
- 横幅補正  
  `85–115` を主条件
- 正方形プロポーション  
  実際の不透明領域の縦横比 `0.80–1.25` を安定域とする

## 5. 論文に書くときの安全な言い方

### 強く言ってよい

> 本研究では、小サイズ表示時の視認性に関わる主要因として、知覚コントラスト、文字の輪郭保持、字形の複雑さ、ならびにフォントウェイトと横幅補正を扱った。

### 慎重に言うべき

> 本研究で提示する推奨値は、既存の可読性研究と APCA 系コントラスト研究を参照しつつ、文字絵文字の制作条件に適用可能な形で構成した設計指針である。

### 断定しないほうがよい

- 「この値が普遍的最適である」
- 「この色が常に最良である」
- 「この枠線幅が全フォント・全背景で最適である」

## 6. 現時点での研究的結論

現時点で最も学術的に筋が良い結論は次である。

1. APCA の厳密実装は可能であり、本システムでは `apca-w3` を用いている
2. カスタム文字絵文字の設計では、色相そのものよりも明度差と外側線の安定性が重要である
3. フォント、ウェイト、横幅補正は小サイズ保持性の候補要因だが、現行の具体的閾値は実験で校正する必要がある
4. 外側線は必須ではなく、実際の背景分離が不足する場合に用いるのが合理的である
5. 黒い内側線は全色に一律適用するのではなく、背景とのコントラスト不足が確認される色の補助として用いるのが合理的である
6. これらは厳密な「最適解」ではなく、視認性を保ちながら表現の自由度も残すための研究上の推奨条件である
